# 相枢日志策略

本文只说明相枢运行事件的日志边界：哪些事件值得记录、写到哪里，以及事件上下文能带到什么程度。
对话投递模型见 `agent-chat.md`。

## 输出位置

游戏进程内日志进入太吾游戏日志系统。前端和后端插件通过 `Wanxiang.Taiwu.Logging` 把消息和结构化上下文
格式化为单行文本，底层仍写入太吾日志。

MCP sidecar 是游戏外进程。它的生命周期、endpoint 注册、宿主进程退出和启动失败等事件写入
`AgentWorkingDirectory/.xiangshu-runtime/Diagnostics/McpServer/` 下的紧凑 JSON 事件日志。

## 游戏进程日志

游戏进程日志用于定位运行边界和失败原因，不承担对话记录职责。正常热键触发、聊天窗口显隐和每轮 CLI 成功
回复不写游戏日志；这些事件已经由界面状态或可见消息体现。

按运行边界分级：

- `Info`：前端/后端 IPC、MCP sidecar 等启动边界；设置更新但需重启生效；手动重置或适配器切换导致的会话
  重建；正在运行的 Agent 调用被中断或重置取消。
- `Warning`：可恢复但会影响功能的边界问题，例如热键 id 冲突、UI 层不可用、资源加载失败或 Agent 设置缺失。
- `Error`：插件初始化失败、MCP sidecar 启动失败、CLI 退出码非零、CLI 回复格式无效、会话快照清理失败或
  异步 UI 任务异常。

CLI 调用失败时，日志可以记录前端会话 id、适配器、CLI 会话模式（`new` 或 `resumed`）、耗时、退出码
（如有）和截断后的 stderr 片段。取消、重置或设置缺失等边界只记录判断该事件所需的最小上下文。

事件上下文只保存定位事实，不保存内容正文。玩家输入、相枢回复、受信脚本源码、完整 stdout/stderr 和外部
Agent 会话 id 原文不进入游戏日志。

前端会话 id 用于把失败归属到相枢自己的本地聊天会话。外部 Agent 会话 id 只用于恢复 CLI 会话，不直接写入
游戏日志；需要区分首次调用和恢复调用时，只记录 `new` 或 `resumed`。

## 共享日志库边界

`Wanxiang.Taiwu.Logging` 只负责把消息、异常和上下文格式化为太吾游戏日志可接收的单行文本。事件选择、字段
取舍和敏感内容边界由调用方所属模块决定。

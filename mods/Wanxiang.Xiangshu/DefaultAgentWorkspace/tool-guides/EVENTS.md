# 事件运行时检索指引

当玩家询问当前正在发生的事件、待处理事件、事件节点、事件选项、选项为何出现或不可选、选项会跳向何处，
或想按事件文字、事件组、事件标识、正文片段、选项文字、触发类型、脚本/条件线索查找事件配置，或分析可能
触发候选时，读取本文件。这里的“事件”指游戏 `TaiwuEvent` 运行时事件；角色的最近经历、生平事件、经历分类
和按经历找角色仍归 `LIFE_RECORDS.md`。

用本文件判断事件事实层、方法锚点、查询路线和解释边界。脚本入口契约、目标侧和线程规则仍归
`RUNTIME_SCRIPTING.md`；稳定机制、配置、本地化、模板显示辅助和百晓册资料仍归 `GAME_KNOWLEDGE.md`。

事件属于后端运行态。检索时先从事件事实层和方法入口下手；返回对象的局部形态只作为当前脚本组织摘要的依据。

## 查询模型

先判断玩家请求属于哪一层事件事实：

- 当前与队列事件：玩家问眼前事件、当前选项、待处理事件、过月事件队列或事件摘要。
- 事件配置库：玩家给出事件标识、事件组、事件名片段、正文片段、选项文字、触发类型或脚本/条件线索，要求定位、
  索引或解析事件节点。
- 可能触发分析：玩家问某类事件是否存在、什么情况下可能触发、哪些事件可能由当前状态或某类触发器引出。

三层事实共用同一套解释规则：玩家当前显示优先读取游戏已经生成的显示数据；事件配置库问题优先读取当前运行时
可检索的配置和脚本结构；可能触发问题返回候选、触发器、条件和结论层级。可见性/可用性求值只适用于带有当前
事件上下文的对象。解释类查询停在只读检索；选项选择、事件跳转和脚本执行都按写操作处理。

## 方法锚点

后端核心域是 `GameData.Domains.TaiwuEvent`。默认从这些方法下手：

- `DomainManager.TaiwuEvent.GetDisplayingEventData()`：读取当前前端显示用事件数据。用于回答眼前事件、可见正文、
  可见选项、可读条件或当前显示状态。
- `DomainManager.TaiwuEvent.GetTriggeredEventSummaryDisplayData()`：读取待处理事件摘要。用于回答队列或待处理事件。
- `DomainManager.TaiwuEvent.GetEvent(guid)`：用当前已确认的事件标识展开已注册事件对象。标识应来自当前显示数据、
  队列摘要、玩家输入或配置库检索结果。
- `DomainManager.TaiwuEvent.GetAllEventConfigs()`：枚举当前运行时注册的事件配置。用于有限搜索、候选分层和结构解析。

事件运行态读取需要后端主线程脚本；具体工具参数按 `RUNTIME_SCRIPTING.md` 和当前工具说明填写。纯文本整理可以在脚本内完成，
但读取 `DomainManager.TaiwuEvent` 仍使用主线程入口。

## 当前事件

默认处理顺序：

1. 在 `backend` 目标侧、`mainThread` 入口调用 `GetDisplayingEventData()`。
2. 让脚本从当前返回对象中提取当前问题需要的最小显示摘要，例如是否存在当前事件、玩家可见正文、可见选项、
   可读条件或可用状态。
3. 玩家追问内部原因时，再用当前显示摘要中的事件标识或选项线索展开事件对象，并只返回条件、消耗、脚本、
   跳转等结构摘要。
4. 需要核对玩家看到的 UI 文本、按钮顺序或玩家是否真的处在事件窗口时，回到 `PLAYER_VIEW.md` 取得玩家视图。

当前事件问题的主路径是显示数据。显示数据不足时，做当前问题相关的小范围只读探测，先确认类型、方法或结构候选，
再组织本次脚本。

## 待处理事件

待处理事件不是当前窗口内容。默认处理顺序：

1. 调用 `GetTriggeredEventSummaryDisplayData()` 获取待处理事件摘要。
2. 摘要只作为展开键和队列事实来源；需要事件组、正文、选项或关联角色名时，再用当前返回的事件标识和角色线索
   继续查询。
3. 结果很多时按队列顺序限量返回；玩家要求完整列表时再继续展开。

处理队列属于写操作；普通检索只读摘要。玩家明确要求处理时，按 `RUNTIME_SCRIPTING.md` 的脚本边界先确认目标事件
和影响范围。

## 配置库检索

这类查询的主语不是当前窗口，而是“当前运行时可检索的事件配置集合 + 搜索谓词”。它是一等检索能力，可用于当前
并未显示的事件。这里的“配置库”指游戏加载并注册事件包后，能通过 `GetEvent(guid)` 和 `GetAllEventConfigs()`
读取到的事件配置；配置存在的结论层级是“可检索”，触发、入队或玩家经历需要来自队列、显示事件或触发流程的证据。

默认处理顺序：

1. 已知事件标识时直接用 `GetEvent(guid)` 展开。
2. 已知事件组、事件名片段、正文片段或选项文字时，遍历 `GetAllEventConfigs()` 做有限字符串匹配。
3. 已知触发类型、事件类型或是否头事件时，用当前返回对象中的对应结构线索过滤；过滤表达式由当前脚本确认。
4. 返回候选摘要而不是完整事件对象；摘要只包含玩家目标需要的标识、组别、匹配来源和少量结构线索。

广域搜索先限量返回候选，并说明还可以按事件组、正文、选项、触发类型、事件类型、是否头事件或脚本/条件线索
继续收窄。

## 节点、选项与脚本结构

默认处理顺序：

1. 先确定目标事件：优先当前显示事件，其次玩家给出的事件标识，最后从搜索候选中选最相关节点。
2. 确定目标选项：优先用当前显示顺序或显示文本；调试语境才使用当前返回对象里的内部选项键。
3. 当前事件选项先以显示数据为准；显示数据不足时，再展开事件对象并读取条件、消耗、脚本和跳转等结构摘要。
4. 解释跳转或脚本影响时只做结构解释；真正选择、跳转或执行脚本属于写操作。

非当前事件通常没有有效事件上下文。这时输出的是结构摘要，例如“有可见条件脚本”“有选中脚本”“会跳到其它事件选项”，
不表述为当前可见或当前可选。

## 可能触发分析

可能触发分析的产物是候选分层。默认处理顺序：

1. 先把玩家问题转成事件配置库搜索谓词，例如事件组、正文/选项文字、触发类型、事件类型、头事件、条件函数或脚本函数。
2. 用 `GetAllEventConfigs()` 找候选事件节点，并返回触发类型、事件类型、条件/脚本摘要、入口脚本和首层跳转线索。
3. 若玩家问题依赖当前世界状态，再读取最小必要运行时事实；条件求值仍以当前运行时上下文是否可用为准。
4. 输出时区分“存在这样的事件配置”“结构上可能由这些条件或跳转引出”“当前状态下已确认会触发”。只有取得队列、
   当前显示事件或触发流程中的明确上下文时，才给出已确认会触发的结论。

脚本函数含义不确定时，先返回函数所属事件和结构位置，再按当前命中的函数做小范围定位；定位细节只服务当前问题。

## 结果解释边界

普通玩家问答优先使用当前显示数据。它最接近玩家实际看到的正文、选项、可用状态和可读条件提示。
事件配置对象是结构事实，不等同于最终显示文本。

解释“为什么不可点”时，优先使用当前显示数据提供的可读原因；这些仍不足时，再说明当前运行时未给出可读原因，
并补充结构事实，例如存在条件、消耗、脚本或跳转。解释“为什么看不到某选项”时，当前显示数据只能证明它未显示；
若要追查原因，需要当前事件对象和有效事件上下文。

需要核对屏幕上的文本、按钮顺序或玩家是否真的处在某个事件窗口时，优先取得玩家视图；后端事件状态只作为权威状态
和原因补充。

## 写入边界

这里只处理事件的检索和解释。选择事件选项、处理待处理事件、跳转事件或改变事件状态都属于写操作，
按 `RUNTIME_SCRIPTING.md` 的脚本边界处理；事件推进的玩家意图和可见收束仍由 `AGENTS.md` 负责。

# GM 与调试入口指引

当前目标需要定位、解释、比较或借用游戏内预置 GM/调试入口，或需要借这些入口寻找比直接写运行时对象更接近游戏已有
逻辑的状态改动路径时，读取本文件。普通游戏机制查询仍先走 `GAME_KNOWLEDGE.md`；当前事件、事件选项和事件脚本结构
仍先走 `EVENTS.md`；实际脚本入口、目标侧、线程规则和写操作边界仍以 `RUNTIME_SCRIPTING.md` 为准。

本文件只提供定位路线、边界和少量高频锚点。GM/调试入口由当前游戏进程中加载的 UI、命令行、方法和事件配置拥有；
完整清单、参数和具体实现以当前运行时事实为准。它们是 Agent 可借用的内部实现路线，不是玩家意图分类，也不是
玩家可见玩法契约；普通玩家答复仍只呈现相枢可见的结果、路径或代价。

检索时先从 GM/调试入口的类型和方法路线下手；枚举结果和本次命中只作为当前查询或当前状态改动的依据。

## 查询模型

先区分当前目标涉及的 GM/调试入口属于哪一类：

- GM 面板与命令行：前端 `UI_GMWindow` 加载带 GM 属性的成员，内置主体在 `GMFunc`，命令行由 `GMCommandLine`
  建立。按钮名、命令名、页面、参数和开关状态以当前枚举结果为准。
- 事件里的 GM 风格选项：某些事件配置带有测试或编辑器痕迹。它们归 `TaiwuEvent` 事件配置和事件脚本结构，
  不等同于 GM 面板命令；是否当前可见以当前显示数据和有效事件上下文为准。

两类入口都可能导向写操作，改变当前运行世界、界面运行态或调试开关。只读检索可以直接做；执行、模拟执行、跳转事件、
改变量、给物品、改人物或跳过流程都按写操作处理。需要改变状态时，本文件用于比较后端域 API、已有 GM/调试入口
和直接运行时改动的所有权、参数边界和验证路径。

## GM 面板路线

GM 面板是前端调试 UI。稳定入口优先从这些类型或方法下手：

- `GameApp`：命令行参数和 GM 面板启用状态的入口。
- `CommonCommandKit.OpenGMPanel`：系统热键入口。玩家是否能看见或操作这个热键，应以玩家视图和当前设置界面为准。
- `GameStateInGame`：进入游戏状态时创建或维护 GM 窗口的入口。
- `UI_GMWindow`：GM 面板 UI、成员扫描、页面构建和按钮调用入口。
- `GMFunc`：内置 GM/调试方法主体。
- `GMCommandLine`：GM 命令行、补全和参数解析入口。
- `GMMemberAttribute`、`GMFuncAttribute`、`GMFuncArgAttribute`、`GMPropertyAttribute`、`GMObjectAttribute`：
  只作为枚举 GM 成员时的类型线索；属性值以当前反射结果为准。

需要确认玩家是否能看见或操作 GM 面板时，先按 `PLAYER_VIEW.md` 取得玩家可见证据。需要枚举、解释、比较或借用
命令路径时，在 `frontend` 目标侧从上述类型做只读枚举；需要改后端世界状态时，优先确认当前世界事实，再决定使用
稳定后端域 API、调用对应 GM 方法，还是只把 GM 实现当作参考来组织更窄的状态改动。

## 事件 GM 选项路线

事件里的 GM 风格选项优先按 `EVENTS.md` 检索：

1. 当前窗口问题先调用 `DomainManager.TaiwuEvent.GetDisplayingEventData()`，确认玩家是否真的看见了 GM 或测试痕迹文本。
2. 非当前窗口的检索用 `DomainManager.TaiwuEvent.GetAllEventConfigs()`，按选项文本、事件组、正文片段或脚本结构找候选。
3. 命中后只把当前目标需要的结构事实返回，例如是否有可见条件、可用条件、选中脚本、跳转或消耗。
4. 如果可见条件结构等价于永假表达式，把结论限定为“配置存在但不是当前正常显示入口”。普通问答只解释结构事实；
   执行等价状态改动属于写操作；当前目标需要跳过、生成、修改或调试状态时，先核对当前事件、奇遇或地图上下文，
   再决定是否借用该选项脚本、复用同一状态改动，或停在结构解释。

## 检索组织

本文件只保留检索路线和高频入口。写当前脚本时按下面的方式组织候选、实现和可执行性判断：

- 枚举 GM 命令：从 `UI_GMWindow`、`GMFunc`、`GMCommandLine` 和 GM 属性类型出发，只返回当前查询需要的候选摘要。
- 检索事件 GM 选项：从 `DomainManager.TaiwuEvent.GetAllEventConfigs()` 出发，只按当前关键词或脚本线索做有限搜索。
- 展开实现：命中目标 GM 方法或事件选项后，再读取当前方法体或事件脚本结构；命中细节只服务当前问题。
- 判断可执行性：先确认当前世界、当前界面和目标上下文；没有有效上下文时停在解释或改用更窄的状态改动。

## 执行边界

- 把 GM 能力当作游戏已有实现的索引和可选执行路径。能用明确后端域 API 完整表达的当前世界
  改动，通常直接用后端域 API 更可控。若目标正好对应已有 GM/调试方法或事件 GM 脚本，先读其实现和参数边界，
  再决定调用该入口、复用同一状态改动，或停在解释。直接改运行时内部状态只用于没有更合适入口，且已经确认
  目标、影响和验证路径的窄范围情形。
- 直接调用 `GMFunc` 方法属于前端写操作，很多方法会走 UI、前端模型、异步桥接或全局静态开关。调用前先确认当前
  游戏状态、目标角色/物品/地点和玩家意图。
- 事件 GM 选项的“等价执行”不一定等同于点击选项；选项可能依赖当前事件上下文、奇遇变量、当前事件、镜头或地图状态。
  当前事件不存在或上下文不匹配时，先只解释结构，或改用更窄的状态改动。
- 玩家只问“有没有 GM 指令、它们是什么”时，停在只读枚举和解释。当前目标需要使用 GM、跳过、生成、修改或调试
  状态时，再按 `RUNTIME_SCRIPTING.md` 的脚本边界执行，并在执行后验证目标结果。
